# Generating a New Copper Project

`cargo-cunew` is the primary Copper scaffolding path.

## Fast path

Install the tool:

```bash
cargo install cargo-cunew
```

Generate a single-crate Copper project backed by the latest stable Copper crates on crates.io:

```bash
cargo cunew my_robot
cd my_robot
cargo run
```

Generate a multi-crate workspace instead:

```bash
cargo cunew --template workspace my_workspace
cd my_workspace
cargo run -p cu_example_app
```

Use a local Copper checkout:

```bash
cargo cunew --source local --copper-root /path/to/copper-rs my_robot
```

Use a git dependency instead of crates.io:

```bash
cargo cunew --source git --git-url https://github.com/copper-project/copper-rs.git my_robot
```

The available templates are:

- `project`: single-crate Copper app (`cu_project`)
- `workspace`: multi-crate workspace (`cu_full`)

The available sources are:

- `crates.io`: default; resolves the latest stable `cu29` and `cu29-export`
- `git`: uses the configured Copper git URL, plus optional `--git-branch`, `--git-tag`, or `--git-rev`
- `local`: points generated manifests at a local Copper checkout

## Direct template usage

The bundled templates still work with `cargo-generate` for template development or direct testing:

```bash
cargo install cargo-generate

# Single-crate template from a repo checkout
cargo +stable generate \
    --path support/cargo_cunew/templates/cu_project \
    --name test_project \
    --destination . \
    --define copper_source=local \
    --define copper_root_path=/absolute/path/to/copper-rs

# Workspace template from a repo checkout
cargo +stable generate \
    --path support/cargo_cunew/templates/cu_full \
    --name test_workspace \
    --destination . \
    --define copper_source=local \
    --define copper_root_path=/absolute/path/to/copper-rs
```

The template-local `justfile` remains useful for repo development:

```bash
cd support/cargo_cunew/templates
just gen-project
just gen-workspace
```

## Utility Commands

The generated project includes helper commands in its `justfile`:

* `just log`: Extract and view structured logs generated by your Copper application.
* `just cl`: Extract CopperLists (sequences of operations) from the application's log output.
* `just resim`: Replay the recorded log once into a fresh replay log.
* `just resim-debug`: Start the replay-backed remote debug server manually.
* `just rcfg`: Generate a `.dot` file representing the application's execution Directed Acyclic Graph (DAG). This currently requires `--source local`. You can use tools like Graphviz to visualize the output (for example, `dot -Tpng graph.dot -o graph.png`).

Set `APP_NAME=test-workspace-demo APP_DIR=test_workspace-demo` to target the demo app (for example, `APP_NAME=test-workspace-demo APP_DIR=test_workspace-demo just log`).

## Replay Target Contract

Generated projects now include a replay binary (`*-resim`) with a consistent CLI:

- `--debug-base <path>`: when present, run the remote debug server instead of a one-shot replay
- `--log-base <path>`: recorded Copper log base to replay
- `--replay-log-base <path>`: replay log output path or per-session template

The scaffolds use CLI arguments only for this contract; they do not depend on environment-variable fallbacks.

To keep replay targets uniform, generated resim binaries build on `cu29::replay`:

- use `ReplayCli` plus `ReplayDefaults` when the replay binary only needs the standard flags
- use `ReplayArgs` with `#[command(flatten)]` when the app has extra CLI such as a mission selector
- use `per_session_replay_log_base(...)` and `serve_remote_debug(...)` instead of open-coding per-session replay/debug plumbing
